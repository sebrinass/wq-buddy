## 数据库字段说明

每条Alpha记录包含以下状态相关字段：

| 字段名 | 类型 | 谁来填 | 何时填 | 示例值 |
|--------|------|--------|--------|--------|
| submit_status | 文本 | 自动+手动 | 回测后/确认后 | 未提交 / 可提交(待查) / 已通过 / 提交失败 |
| correlation_max | 数字 | **手动** | 用户查平台后 | 0.9696 |
| correlation_min | 数字 | **手动** | 用户查平台后 | -0.0687 |
| reject_reason | 文本 | **手动** | 失败时 | "Self-correlation > 0.7..." |

**为什么 correlation 是手动填写？**
- 平台回测结果不直接返回相关性数值
- 需要用户在平台上手动点击查看Self Correlation
- AI无法自动获取，必须等用户告知

## 4种状态定义与转换

```
未提交 ← IS Checks有FAIL / 回测失败
  ↓
  IS全通过
  ↓
可提交(待查) ← 自动判断，等待用户确认
  ↙           ↘
用户确认OK    用户告知失败
  ↓              ↓
已通过          提交失败
```

状态速查详见 SKILL.md "提交状态工作流"章节

## 状态详细说明

### 1. 未提交

- **触发条件**：回测失败 / IS Checks有FAIL项 / 指标太差
- **需要用户操作**：无
- **AI动作**：显示结果，不提醒后续操作

### 2. 可提交(待查) — 重点监控状态

- **触发条件**：回测成功 + IS Checks全部PASS + SELF_CORRELATION仍PENDING
- **需要用户操作**：**必须去平台查看Self Correlation**
- **AI动作**：**强制提醒**（见SKILL.md闭环规则）
- **用户返回后转换为**：已通过 or 提交失败

### 3. 已通过

- **触发条件**：用户确认相关性OK
- **需要用户操作**：告诉AI "通过了" 或 "可以提交"
- **AI动作**：`db.updateSubmitStatus(id, '已通过')`，同时记录correlation_max/min

### 4. 提交失败

- **触发条件**：用户告知不能提交及原因
- **需要用户操作**：提供reject_reason内容
- **AI动作**：`db.updateSubmitStatus(id, '提交失败', reason)` + `db.updateCorrelation(id, max, min)`

## 特殊场景说明

### 批量场景处理

```
场景: 10个Alpha回测完，8个显示"可提交"

步骤1: AI一次性列出所有可提交的Alpha
步骤2: 用户响应类型:
  A) 逐个回复 → AI逐个更新状态
  B) "先记着/稍后查" → AI设置提醒
  C) "都通过/都失败" → 批量更新
```

### 英文术语对照

| 英文 | 中文 | 说明 |
|------|------|------|
| SELF_CORRELATION | 自相关性 | 和自己的历史表现对比 |
| PENDING | 待定 | 平台还在计算中 |
| PASS | 通过 | 相关性低，可以提交 |
| FAIL | 失败 | 相关性太高，被拒 |

## 闭环检查清单

**每次回测后必须逐项检查：**

- 结果已用中文格式化显示
- 检查submit_status是否为"可提交(待查)"
- 检查correlation_max/min是否为空
- 检查reject_reason是否需要填写
- 如有可提交Alpha，已提醒用户查平台
- 等待用户响应后才结束本次任务

闭环规则详见 SKILL.md 第2章"闭环规则"
